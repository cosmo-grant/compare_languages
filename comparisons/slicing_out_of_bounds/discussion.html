John Ousterhout recommends "defining errors out of existence":

The best way to eliminate exception handling complexity is to define your APIs so that there are no exceptions to handle: define errors out of existence.
— A Philosophy of Software Design, p81

One of his examples is slicing, as here. If you define slicing as "get the item at index i for start <= i < end", you must error for out of bounds indexes. But if you define it as "get any items with indexes at least startand less than end", you needn't.

Compare looking up non existent keys.

I remembered Ousterhout's advice when writing a function to get how long since a task had started. What should the function return if the task hadn't started yet? I originally had it throw an exception. But it was better to follow Ousterhout's advice: instead of asking "how long since the task started?" and throwing an exception, ask "how long has the task been running?" and return 0.
