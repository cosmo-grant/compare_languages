What happens when you look up a non-existent key via the typical [] syntax?

In Python, it raises an exception. In Go, you get the value type's zero value. In Ruby, you get nil.

John Ousterhout recommends defining errors out of existence:

The best way to eliminate exception handling complexity is to define your APIs so that there are no exceptions to handle: define errors out of existence.
— A Philosophy of Software Design, p81

Key lookup is an example. If you define lookup as "get the value for the key", you must raise an exception when no such key. If you define it as "get the value for the key if the key exists, else a fallback", you needn't.

I agree with the examples Ousterhout talks about in the book. But I'm not sure about key lookup. Don't you typically need to know whether the value you got is just the fallback?

Sure, in Go you can check that via comma ok (v, ok = m["bar"], where ok flags if the key is in the map), and in Ruby via h.has_key?("bar"), then branch accordingly. My point is that then you don't end up with simpler code. A try/except in Python becomes an if in Go and Ruby.

Besides, Python provides d.get() for safe lookups if that's what you want. (Perhaps Ousterhout would reply, "Python wouldn't need a second method if it had defined the first one right".)

It's interesting that Python's Counter type, a dict subclass, returns 0 for missing keys. 

from collections import Counter
c = Counter(["foo", "bar", "foo"])
print(c["baz"])  # 0

And that in Python [] errors but the supplementary get() returns a fallback, in Ruby [] returns a fallback and the supplementary fetch() errors.
